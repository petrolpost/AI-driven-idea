## 📖 技术文章分析规则

### 1. 文章阅读与分析

**阅读策略**:

- 🎯 **目标导向**: 重点关注可实现的技术方案、决策框架、工具方法
- 🔍 **深度理解**: 分析文章的核心观点、技术原理、应用场景
- 💡 **价值提取**: 识别具有实用价值的概念、算法、评估方法
- 🚀 **可行性评估**: 判断技术方案的实现难度和实用性

**分析维度**:

- **技术深度**: 是否包含具体的技术实现细节
- **实用价值**: 能否解决实际问题或提升工作效率
- **创新性**: 是否提供新的思路或改进现有方案
- **可扩展性**: 是否具备进一步开发和优化的潜力

### 2. 工具项目识别标准

**合格项目特征**:

- ✅ **明确需求**: 有清晰的问题定义和解决目标
- ✅ **技术可行**: 基于现有技术栈可以实现
- ✅ **实用价值**: 能够提供实际的业务或开发价值
- ✅ **独立性**: 可以作为独立项目进行开发和维护

**特殊情况处理**:

- 📄 **纯理论文章**: 当技术文章中没有提取出具体可用的工具项目时，仅需创建项目简介文档（如README.md），记录文章的核心观点、技术洞察和理论价值，无需进一步的开发组织和代码实现
- 📚 **知识归档**: 此类项目主要用于知识管理和技术积累，为后续相关项目提供理论基础和参考资料
- 🔗 **关联引用**: 在后续开发相关工具项目时，可以引用这些理论基础项目作为技术背景

**项目类型分类**:

- 🔧 **工具类**: 提升开发效率的实用工具
- 📊 **分析类**: 数据分析、决策支持系统
- 🤖 **AI应用**: 基于AI技术的智能应用
- 🌐 **Web服务**: 提供在线服务的Web应用
- 📚 **框架类**: 可复用的开发框架或库

### 3. 工具评估与决策流程

> 在识别出潜在工具项目后，需要进行系统性评估，为用户决策提供数据支持。

#### 📊 工具实用性评估

**评估维度**:

- 🎯 **问题解决度** (1-10分)
  - 解决问题的核心程度
  - 替代方案的可用性
  - 痛点缓解效果

- 💼 **市场需求度** (1-10分)
  - 目标用户群体规模
  - 使用频率预期
  - 商业价值潜力

- 🔧 **技术创新度** (1-10分)
  - 技术方案的新颖性
  - 与现有工具的差异化
  - 技术领先性

- 🚀 **易用性指数** (1-10分)
  - 学习曲线陡峭程度
  - 部署复杂度
  - 用户体验友好度

**评估方法**:

```markdown
## 工具评估报告

### 工具概述
- **工具名称**: [工具名称]
- **核心功能**: [简述主要功能]
- **目标用户**: [用户群体描述]

### 实用性评分
- 问题解决度: X/10 - [评分理由]
- 市场需求度: X/10 - [评分理由]
- 技术创新度: X/10 - [评分理由]
- 易用性指数: X/10 - [评分理由]
- **综合评分**: XX/40
```

#### 🏗️ 开发复杂度分析

**复杂度等级**:

- 🟢 **简单** (1-3周)
  - 单一功能模块
  - 基础CRUD操作
  - 简单数据处理
  - 标准技术栈实现

- 🟡 **中等** (1-2个月)
  - 多模块集成
  - 复杂业务逻辑
  - 第三方API集成
  - 数据库设计优化

- 🟠 **复杂** (2-4个月)
  - 分布式架构
  - 机器学习集成
  - 实时数据处理
  - 高并发性能优化

- 🔴 **极复杂** (4个月以上)
  - 大规模系统架构
  - 多技术栈融合
  - 算法研发需求
  - 企业级安全要求

**复杂度评估因子**:

- 📋 **功能复杂度**: 功能模块数量和交互复杂性
- 🔧 **技术复杂度**: 技术栈难度和集成复杂性
- 📊 **数据复杂度**: 数据处理量和算法复杂性
- 🔒 **安全复杂度**: 安全要求和合规性需求
- 🎨 **UI/UX复杂度**: 界面设计和交互复杂性

## 🏗️ 项目组织结构规则

### 1. 目录结构标准

```
项目根目录/
├── 项目导航.md                 # 项目索引文件
├── .trae/
│   └── rules/
│       └── project_rules.md    # 本规则文件
└── [项目名称]/                 # 具体项目目录
    ├── README.md               # 项目说明文档
    ├── pyproject.toml          # Python项目配置
    ├── .python-version         # Python版本指定
    ├── Makefile               # 开发命令集合
    ├── .gitignore             # Git忽略规则
    ├── .env.example           # 环境变量示例
    ├── main.py                # 主程序入口
    ├── run.py                 # 运行脚本
    ├── docs/                  # 文档目录
    ├── examples/              # 示例代码
    ├── tests/                 # 测试代码
    ├── static/                # 静态资源
    └── assets/                # 项目资源
```

### 2. 命名规范

**项目命名**:

- 使用小写字母和连字符: `ai-decision-framework`
- 体现项目核心功能: `text-analyzer`, `api-gateway`
- 避免过长或过于抽象的名称

**文件命名**:

- Python文件: 小写字母+下划线 `decision_engine.py`
- 文档文件: 小写字母+连字符 `case-studies.md`
- 配置文件: 标准名称 `pyproject.toml`, `Makefile`

### 3. 项目导航维护

**更新时机**:

- ✅ 新项目创建时立即更新
- 🔄 项目状态变更时及时同步
- 📊 定期检查和维护统计信息

**更新内容**:

- 项目概览表格
- 详细项目信息
- 统计数据
- 更新日志

**项目链接规范**:

- 📎 **项目名称链接**: 项目概览表格中的项目名称必须链接到对应项目目录下的 `README.md`文件
- 🔗 **链接格式**: 使用相对路径格式 `[项目名称](项目目录/README.md)`
- ✅ **链接验证**: 确保所有项目链接指向正确的README文件路径
- 🔄 **链接维护**: 项目目录变更时同步更新导航文件中的链接
